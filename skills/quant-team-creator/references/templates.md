## Team-Lead Control Plane

- team-lead = the main conversation, not a spawned agent
- team-lead owns user alignment, scope control, task decomposition, phase transitions
- team-lead maintains project-global truth: main `task_plan.md`, `decisions.md`, and this `CLAUDE.md`
- team-lead enforces the two mandatory quant gates:
  - **research-validator gate**: no research output (feature / model / signal) moves downstream without [OK]
  - **risk-manager gate**: no strategy goes live without [RISK-OK]
- team-lead decides whether a workflow improvement is project-local or should be written back into `CC_quant_team`
- **No standalone subagents**: Once the team exists, ALL work goes through teammates via SendMessage. Only exception: spawning a new teammate (with `team_name`) to join the team

### Team-Lead Bookkeeping (MANDATORY — survives compaction via this CLAUDE.md)

**Team-lead owns 5 root files: task_plan.md / progress.md / findings.md / decisions.md / team-snapshot.md.** Agents write to their own dirs; without team-lead writing to root, no project-level record exists.

| Event | task_plan.md | progress.md | findings.md | decisions.md | team-snapshot.md |
|-------|:---:|:---:|:---:|:---:|:---:|
| Dispatch a task | §4: add row | append | — | — | — |
| Agent completes | §4: mark done | append + verdict | — | — | — |
| Validator verdict | §6: gate status | append | — | — | — |
| Architecture decision | — | append | — | new D<N> | — |
| **Phase boundary / Milestone** | §5: phase status | phase summary | **consolidate** | if applicable | check staleness |
| Roster change | — | append | — | — | **regenerate** |
| Session resume | — | status summary | — | — | check staleness |

**Milestone Consolidation** (at phase boundaries or significant completions):
1. Read agent task-folder findings → extract key results, metrics, conclusions → write consolidated `[MILESTONE]` entry to root `findings.md`
2. Extract durable knowledge (schemas, metrics, interfaces, capacity numbers) → create or update the relevant `docs/` file (data-schemas.md / pipeline-flow.md / strategy-contracts.md / invariants.md)
3. Update `docs/index.md` if any docs/ file changed

This is NOT copying verbatim — team-lead synthesizes scattered agent findings into structured docs/ knowledge that downstream agents can consume from one place.

**Self-check (every ~5 interactions):** task_plan.md §4+§5+§6 current? progress.md has last dispatch/completion? findings.md consolidated since last agent completion? docs/ up to date? team-snapshot.md fresh? If no → update NOW.

## Task Dispatch Protocol

### Message Delivery Timing (Critical)
`SendMessage` lands only when the recipient is idle — you cannot interrupt a running task. Front-load context in the initial dispatch, no mid-task additions. Read `progress.md` / `findings.md` directly for live status (**files live, messages not**). Full guide: SKILL.md § Message Delivery Timing Constraint.

### TaskCreate Description Format

TaskCreate description: one-line scope + acceptance criteria + `.plans/` path.
Example: `"Build orderflow imbalance feature family. Input: data-engineer's schemas at docs/data-schemas.md §L2. Output: feature parquet + findings.md. See .plans/x/hft-researcher-orderflow/research-orderflow-imbalance/task_plan.md"`
Assign owner and dependencies via TaskUpdate. Teammates self-serve unblocked tasks via TaskList.

### Large Tasks — STOP AND CHECK BEFORE SENDING

**Before sending a large task to any agent, verify your message includes ALL 5 items below:**

1. **Scope & goals**: What to build, acceptance criteria
2. **Document reminder**: "Create a task folder `<prefix>-<task-name>/` with task_plan.md + findings.md + progress.md, and add an index entry to your root findings.md"
3. **Dependencies**: Which research/tasks they depend on, key file paths and line numbers
4. **Review expectation**: Whether code-validator / research-validator / risk-manager review is needed after completion
5. **Schema update expectation**: If output changes any data/model/strategy schema, state which `docs/*.md` file must be updated

Example:
```
SendMessage(to: "hft-researcher-orderflow", message:
  "New task: orderflow imbalance feature family.
   Scope: top-of-book imbalance + trade-sign imbalance + queue-position features.
   Depends on: data-engineer's OrgData schema at .plans/<project>/docs/data-schemas.md §OrgData
   Output: feature parquet + findings.md with feature output schemas
   Please create research-orderflow-imbalance/ folder and update your root findings.md index.
   Schema update: add feature output schemas to docs/data-schemas.md §Features
   This is a large feature family — request research-validator after completion.")
```

## Harness Checklist

Team-lead reviews at phase boundaries (not every task):

### Operational Health (every phase boundary)

- **Docs harness**: Read CLAUDE.md + main task_plan.md — still accurate? If stale → update before dispatching next phase
- **Gate harness**: Every research output in this phase passed research-validator [OK]? Every strategy passed risk-manager [RISK-OK]?
- **Observability harness**: Grep progress.md for "error|fail" — are failures logged with enough detail?
- **Invariant harness**: Review Known Pitfalls below — should any entry become a reviewer checklist item or an automated Q-N check?
- **Reproducibility harness**: Random spot-check one model artifact — can it actually be reproduced from saved metadata?
- **Replay harness**: Did this phase produce a reusable pattern? If so, note with `[TEAM-PROTOCOL]` for future reference

### Assumption Audit (model upgrade or project retro)

> Every harness component encodes an assumption about model limitations.
> Audit when: new model release, project retro, or a mechanism consistently adds no value.

| Component | Assumption | Still load-bearing? | Action |
|-----------|-----------|--------------------:|--------|
| Task folders | Model loses coherence without decomposition | Did agents use folders effectively? | Keep / Simplify / Remove |
| 3-Strike protocol | Model retries infinitely without guardrails | How often did 3-Strike actually trigger? | Keep / Raise threshold / Remove |
| Context recovery protocol | Model cannot resume after compaction without explicit state reload | Does model pick up context naturally? | Keep / Simplify / Remove |
| Research-validator gate | Model cannot self-audit for subtle bias | Did research-validator catch real bugs, or rubber-stamp? | Keep / Reduce to spot-check |
| Risk-manager gate | Model underestimates tail risk when eager to deploy | Did risk-manager block any strategy that would have lost money live? | Keep / Reduce / Remove |
| Reproducibility gate | Model forgets to version artifacts | Were models actually reproducible when tested? | Keep / Simplify |
| CI gate before review | Model skips verification without enforcement | Did devs run CI voluntarily? | Keep / Trust dev |

**Decision rule**: If a component triggered <2 times in the last phase AND removing it wouldn't have caused observable quality drop → candidate for removal. Record decision in `decisions.md`.

## Style Decisions

> User taste preferences captured during the project.
> When a pattern appears 3+ times, add a new Q-N check to `quant_golden_rules.py`.
> Format: decision, source, enforcement status.

| # | Decision | Source | Status |
|---|----------|--------|--------|
| SD-1 | 2-space indentation, 120-column line max | User global CLAUDE.md | Manual |
| SD-2 | Google-style docstrings (Args/Returns/Raises) on public functions | User global CLAUDE.md | Manual |

- `Manual` — documented only, code-validator checks by convention

### Task Dispatch: Minimize Information Loss

Inter-agent messages lose detail. Every task dispatch MUST be self-contained:
- Reference file paths to findings/docs (agent reads the file, not your summary)
- Include acceptance criteria (agent knows when it's done)
- For research tasks: specify data range, OOS cutoff, feature output schema location

### Template-Level vs Project-Local Changes

Use this split:

- **Project-local**: only this project's docs/workflow need the change
- **Template-level**: future quant teams should inherit the change → update `CC_quant_team` first

Typical template-level changes:

- team-lead responsibilities
- role boundaries
- onboarding protocol
- CLAUDE.md structure
- task/finding/progress conventions
- rebuild timing rules
- new Q-N checks that apply to any quant project

## 3. Phases Overview

Task dispatch is managed via native TaskCreate/TaskList (dependencies auto-unblock).

### Slicing Principle

Break tasks into **vertical slices**: each slice delivers a narrow but COMPLETE path through the pipeline (e.g., one feature family's: schema → computation → validation → test → doc).

### Phases (Standard Quant Flow)

- **Phase 0: Requirements Alignment** — team-lead aligns with user on data sources, frequency focus, success criteria
- **Phase 1: Data Engineering** — data-engineer builds / validates pipeline
- **Phase 2: Feature Research** — hft/lft-researcher(s) produce features, each passing research-validator
- **Phase 3: Model Research** — model-researcher combines features into signals; each model passes research-validator
- **Phase 4: Strategy Development** — strategy-researcher produces backtests; each strategy passes risk-manager
- **Phase 5: (Live) Deployment** — backend-dev integrates; frontend-dev builds monitoring; risk-manager sets circuit breakers

Phases run partially in parallel where independent (e.g., hft-researcher and lft-researcher may work concurrently once data schemas are stable).
